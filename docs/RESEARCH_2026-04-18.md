## #7 Teleport / fast-travel — new data

**Source**: bbfox0703 `CrimsonDesert.CT` entries 174/175/176 ("Blink spell" set, CE 7.6+).

### (a) worldOffset global — world-space ↔ entity-local conversion

Every position on an entity is stored in world-offset-relative space.
The global `worldOffset` Vec4 is subtracted from the raw world position
before any entity writes it to its own struct. To move an entity in
world space you must account for this offset.

- **AOB** (module `CrimsonDesert.exe`):
  `0F ?? ?? ?? ?? ?? ?? 0F 11 ?? 90 00 00 00 E8 ?? ?? ?? ?? F3`
- **RIP-relative resolve**: `pos=3`, `aoblen=7` → the 4 bytes at
  `(match+3)` are a signed 32-bit displacement to the Vec4 global.
- **Confirmed use site**: `CrimsonDesert.exe+0x278C6C8`
  ```
  +278C6C8  movlhps  xmm3,xmm2                       ; xmm3 = {x,y,z,w} raw world pos
  +278C6CB  subps    xmm3,[worldOffset_addr]         ; subtract global
  +278C6D2  movups   [rcx+0x90],xmm3                 ; write entity-local pos
  ```
- **Status**: Community-reported, single source (bbfox0703).

### (b) Entity velocity write (moving-player hook point)

When the player is moving under their own power, velocity ends up written
to `[entity+0x1B0]` as a packed `{X, Z(height), Y, W}` float4 (Z is lane 1,
NOT lane 2 — this is a Black Desert Online legacy convention).

- **Injection AOB** (unique 17-byte pattern):
  `0F C6 ?? 00 0F 59 ?? ?? 8B ?? 0F 11 ?? B0 01 00 00`
- **Injection point**: `CrimsonDesert.exe+0x2791A16`
  ```
  +2791A16  shufps  xmm1,xmm1,00            ; broadcast velocity scalar
  +2791A1A  mulps   xmm1,xmm6               ; apply entity axis weights
  +2791A1D  mov     rax,[rsi]               ; rsi = entity slot, rax = entity
  +2791A20  movups  [rax+0x1B0],xmm1        ; write velocity
  ```
- **Corollary offsets** from the same function:
  - `[rdi+0x270]` — 1-byte flag
  - `[rdi+0x278]` — float accumulator (delta-time scaled)
  - `[rdi+0x27C]` — incrementing counter (tick)
  - `[rdi+0x358]` — float (derived product)
  - `[rdi+0x47C]` — dword (state)
- **Status**: Community-reported, single source.

### (c) Map waypoint / fast-travel trigger capture

When the game propagates a map waypoint (user-selected fast-travel
destination) to an entity, it is copied into:
- `[r14+0xD8]` as a **double** (`movsd`) — packs X and Y as two 32-bit
  floats in one 64-bit write (little-endian: X at +0xD8, Y at +0xDC)
- `[r14+0xE0]` as a **dword** — Z (float)
- `[r14+0x108]` as a qword pointer (origin / source struct)
- `[r14+0x1E6]` as a byte (flag)

- **Injection AOB** (20-byte unique):
  `F2 41 0F 11 86 D8 00 00 00 ?? ?? ?? ?? 41 89 86 E0 00 00 00`
- **Injection point**: `CrimsonDesert.exe+0xAB5594`
  ```
  +AB5594  movsd  [r14+0xD8],xmm0          ; xmm0 sourced from [r15+0x1C] = waypoint XY
  +AB559D  mov    eax,[r15+0x24]           ; waypoint Z
  +AB55A1  mov    [r14+0xE0],eax
  ```
- **Source-struct layout** (r15 during the hook):
  - `r15+0x00` (word) — waypoint type id
  - `r15+0x08` (qword) — parent struct pointer
  - `r15+0x10` (byte)  — flag
  - `r15+0x18` (dword) — stat
  - `r15+0x1C` (double = X float + Y float packed)
  - `r15+0x24` (float) — Z
  - `r15+0x28` (qword) — aux pointer
  - `r15+0x48` (byte)  — flag
- **Status**: Community-reported, single source. Solid — this is the
  exact hook we would need to synchronize fast-travel targets in co-op.

### Implication for co-op

To sync fast travel between peers we can hook `+0xAB5594`, snapshot
`[r15+0x1C..0x28]` (waypoint coords + type), broadcast to the peer, and
on the receiver side write the same values into their own r14 slot (or
call the waypoint apply function — entry unknown, but the hook site
gives us both the inputs and one valid apply-path). Implementation is a
follow-up; this research just makes it tractable.

## #5 Dragon HP — narrowed candidate

**Source**: bbfox0703 CT entry 160 ("Dragon mount timer") injection
context at `CrimsonDesert.exe+0x339D8CB`.

The dragon-mount struct is written field-by-field in a single procedure
ending at the timer write. Reading the surrounding code gives a
contiguous field map for `r13 = dragon_mount*`:

| Offset | Size | Written as | Likely meaning |
|--------|------|------------|----------------|
| `+0xA0` | 16B | `movups xmm6` | Quat or Vec4 (rotation) |
| `+0xB0` | 16B | `movups xmm0` (from [rbp-0x38]) | Vec4 |
| `+0xC0` | 8B  | `movsd xmm0` (from [rax]) | World X+Y packed |
| `+0xC8` | 4B  | `mov eax,[rax+0x08]` | Stat dword |
| **`+0xD8`** | **4B** | **`movss xmm8`** | **Strongest HP candidate** |
| `+0xE0` | 4B  | `mov eax,[rbp-0x7C]` | Stat dword |
| `+0xE4` | 4B  | `mov eax,[rbp-0x78]` | Stat dword |
| `+0xE8` | 4B  | `mov eax,[rbp-0x74]` | Stat dword |
| `+0xEC` | 4B  | `mov eax,[rbp-0x70]` | Stat dword |
| `+0xF8` | 8B  | `mov rax,[rcx+0x08]` | Sub-object ptr |
| `+0x100`| 8B  | `mov rax,[rcx]` | Parent/self ptr |
| `+0x108`| 4B  | dword | Stat |
| `+0x110`| 8B  | qword ptr | Sub-object ptr |
| `+0x118`| 8B  | qword ptr | Sub-object ptr |
| `+0x120`| 4B  | dword | Stat |
| `+0x15C`| 4B  | `= 0xFFFFFFFF` | Reset/invalid marker |
| `+0x160`| 4B  | dword | Dragon flight/ability timer (KNOWN) |

- **Rationale for +0xD8 as HP**: xmm8 is conventionally used for
  high-register float stats on the mount init path (same pattern as the
  horse HP write path). `+0xC0..+0xCC` block is clearly coord/stat, and
  `+0xE0..+0xEC` is a cluster of 4 dwords (likely a stat quad). `+0xD8`
  sits in between and is the only standalone 4-byte float.
- **Status**: Community-reported, single source. Needs in-game read-back
  confirmation (does `[dragon_mount+0xD8]` match the visible HP bar?).

### Actionable narrowing of the dynamic scan

The current scan in `Mount::DRAGON_HP_SCAN_MIN_OFFSET..MAX_OFFSET`
(`0x08..0x200`, stride `0x04`, range `100.0..1e7`) is conservative. We
can tighten it to prefer `+0xD8` and only fall back to a wider scan if
that float fails the plausibility check. See the proposed `DRAGON_HP_
PREFERRED_OFFSET` constant in the Integration section below.

## #4 Camera struct — confirmed negative (no runtime fields beyond +0xD8)

Surveyed:
- **UltimateCameraMod** (FitzDegenhub) — pure PAZ/XML, explicitly
  states "No DLL injection, no memory hacking". 150+ camera states
  live in `playercamerapreset.xml` inside `0010/0.paz`.
- **CDCamera / Proper 3rd Person / Cinematic Vision** — all nexus
  listings are 403 from automated tooling, but the CDUMM metadata and
  GitHub mirrors that are reachable describe these as XML/PAZ mods,
  not runtime hooks.
- **bbfox0703 CT** — no camera entries at all.

**Conclusion**: the full camera struct around `+0xD8` is not publicly
mapped because no public mod needs it — the community modifies camera
through XML presets, not runtime writes. Unless we do our own CE pass
on the captured `camera_struct_ptr` (hooks.cpp:500), this stays at
"partial (zoom only)". Existing `Camera` namespace comment at
`include/cdcoop/core/game_structures.h:331-363` already documents this
accurately — no changes needed.

## Cross-reference: character HP slot offsets

bbfox CT entries 57/61/65 list three Player_Base_addr pointer chains
for Kliff / Damiane / Oongka that differ only in one slot offset:

| Character | Slot offset (in chain) |
|-----------|------------------------|
| Kliff     | `0xD0` |
| Damiane / Oongka (#1.2) | `0xD8` |
| (#1.3) | `0xE0` |

`docs/REVERSE_ENGINEERING.md:109-114` currently lists slot offsets as
Kliff=`0x68`, Oongka=`0xE0`, Damiane=`0x168`. These two numbering
schemes measure from different roots — existing table is from the
player body-slot array (actor +0xD0..+0x108), bbfox uses a deeper
pointer chain. Both are internally consistent; no update needed.

the new constants belong in `include/cdcoop/core/game_structures.h`:

```cpp

    // Map waypoint apply — where the game copies a fast-travel target
    // into an entity struct. r14 = entity, r15 = source struct.
    // AOB: F2 41 0F 11 86 D8 00 00 00 ?? ?? ?? ?? 41 89 86 E0 00 00 00
    constexpr uint32_t WAYPOINT_APPLY_RVA = 0xAB5594;
    constexpr uint32_t ENTITY_WAYPOINT_XY = 0xD8;  // double (X, Y packed)

    constexpr uint32_t SRC_WAYPOINT_XY    = 0x1C;  // double on [r15]
    constexpr uint32_t SRC_WAYPOINT_Z     = 0x24;  // float on [r15]

    // World-offset global — subtract from raw world pos to get entity-
    // local pos at [entity+0x90]. Resolved via RIP-relative AOB.
    // AOB: 0F ?? ?? ?? ?? ?? ?? 0F 11 ?? 90 00 00 00 E8 ?? ?? ?? ?? F3
    //      (pos=3, len=7)
    constexpr uint32_t WORLD_OFFSET_USE_RVA = 0x278C6C8;

    // Entity velocity write. Velocity is a {X, Z_height, Y, W} packed
    // float4 at [entity+0x1B0].
```
